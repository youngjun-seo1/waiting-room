# Waiting Room - 발표 자료

## 리더 미팅 소개 (3~5분)

---

## 이런 상황, 겪어보셨나요?

- 쿠폰 선착순 이벤트 오픈 → 순간 트래픽 폭주
- 서버 다운 → 사용자 이탈 → CS 폭주
- "새로고침 전쟁" → 누가 먼저 들어갔는지 알 수 없음

**해결책: 대기실(Waiting Room)**

> 놀이공원 줄서기처럼, 서버에 들어갈 수 있는 인원을 제한하고
> 나머지는 순서대로 대기시키는 시스템

---

## Waiting Room이 하는 일

```
사용자 접속
    ↓
[대기실] 자리 있어? ──Yes──→ [서비스 페이지] (오리진 서버)
    ↓ No
[대기 페이지]
  "현재 32번째, 약 2분 대기"
    ↓ (자동 입장)
[서비스 페이지]
```

- 선착순 FIFO 대기열 (공정한 순서 보장)
- 실시간 순번/대기시간 표시 (SSE)
- 자리 나면 자동 입장 (새로고침 불필요)

---

## 어떻게 쓰나요?

### 기존 서비스에 코드 변경 없이 적용

```
[변경 전]  사용자 → 서비스 서버
[변경 후]  사용자 → Waiting Room → 서비스 서버
```

- 리버스 프록시 방식 — 서비스 서버 앞에 놓기만 하면 됨
- 기존 서비스 코드 수정 불필요

### 스케줄 기반 자동 운영

- 이벤트 시작/종료 시간 설정 → 자동으로 대기실 ON/OFF
- 수동 개입 없이 운영 가능

### Admin 대시보드

- 실시간 모니터링 (활성 사용자, 대기열 현황)
- 스케줄 등록/관리
- Quick Test로 즉시 테스트 가능 (Small/Normal/Large 프리셋)

---

## 실제 동작 화면

### 대기 페이지 (사용자가 보는 화면)
- 현재 순번, 예상 대기시간, 진행률 바
- 입장 가능 시 자동 리다이렉트

### Admin 대시보드 (운영자 화면)
- ON AIR / STANDBY 상태 표시
- 큐 플로우 애니메이션 (대기 → 입장 → 활성 → 퇴장)
- 스케줄별 통계 (Peak Active, Total Admitted 등)

## 성능

**서버 4대 + Redis 구성**

| 동시접속 | 에러 | 처리량 | 서버 리소스 (서버당) | Redis 리소스 |
|---------|------|--------|-------------------|-------------|
| 1,000명 | 0건 | 3,257 req/s | CPU 9%, 메모리 20MB | 8K ops/s, 37MB |
| 5,000명 | 0건 | 3,872 req/s | CPU 42%, 메모리 31MB | 40K ops/s, 37MB |
| 10,000명 | 0건 | 3,734 req/s | CPU 40%, 메모리 31MB | 49K ops/s, 37MB |

- 10,000명 동접까지 에러 0건
- 서버당 메모리 31MB — 매우 경량
- Redis 연결만으로 멀티 서버 수평 확장

## 기술 스택 (참고)

- **Rust** (axum) — 고성능 리버스 프록시
- **Redis** — 멀티 서버 상태 공유
- **React + TypeScript** — Admin 대시보드

## 정리

1. **서비스 코드 변경 없이** 앞에 놓기만 하면 동작
2. **스케줄 설정**으로 이벤트 자동 운영
3. **공정한 선착순** + 실시간 대기 안내
4. **가볍고 빠름** — 서버 1대, 메모리 31MB로 1,000명 동접
5. **비용 효율** — Cloudflare 대비 월 $200+ 절감
